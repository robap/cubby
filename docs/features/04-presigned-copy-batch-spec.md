# Presigned URLs, CopyObject & DeleteObjects — spec

**Status:** done · **Roadmap:** Phase 4 · **Slug:** 04-presigned-copy-batch

## Why

Two different client behaviors converge in this phase, both invisible until a
real SDK exercises them:

1. **Presigned (query-string) auth.** SDKs generate a short-lived URL that
   carries the whole SigV4 signature in the query string
   (`X-Amz-Algorithm`, `X-Amz-Credential`, `X-Amz-Signature`, `X-Amz-Expires`,
   …) so a browser, a `curl`, or a service with no credentials can `GET` or
   `PUT` one object. This is *the* pattern behind "upload straight from the
   frontend" and "email someone a download link." Per CONCEPT it comes
   **mostly free via `s3s`** — which owns SigV4 for both header and query-string
   auth — so the deliverable here is **proving it**, not building it.

2. **The two object ops SDKs reach for constantly but Phases 1–3 skipped:**
   - **CopyObject** — `s3.copy()`, `aws s3 cp s3://a s3://b`, and the
     **source==dest metadata-only update** idiom every SDK uses to change an
     object's content-type or user metadata without re-uploading bytes.
   - **DeleteObjects** (batch) — `s3.delete_objects()` and the engine behind
     `aws s3 rm --recursive` / `rclone delete`, which delete up to 1000 keys in
     one request instead of N round-trips.

North stars served:
- **Compatibility is proven, not claimed** — acceptance is real presigned URLs
  generated by boto3 / aws-sdk-js v3 / the AWS CLI *resolving against the
  running server*, and real `copy_object` / `delete_objects` SDK calls — never
  hand-rolled requests.
- **The filesystem is the API too** — a copied object is a **real file** at
  `buckets/<destb>/<destkey>` (`cmp` against the source is clean); a batch
  delete leaves the named files **gone** on disk.
- **Dev tool first** — presigned URLs are what people paste into the live log
  and screenshot (Phase 5); getting the auth surface right now is what makes
  that demo honest.

## In scope

- **Presigned / query-string SigV4 verification.** Confirm the existing
  `s3s` + `SimpleAuth` wiring accepts a valid query-string-signed `GET` and
  `PUT` (and rejects a tampered/expired one) with **no new auth code** beyond
  what Phases 1–3 already stand up. If a gap is found, close it minimally.
- **CopyObject** (`PUT /<destbucket>/<destkey>` with `x-amz-copy-source`):
  - Server-side copy of an existing object's bytes into a new key, streamed
    through the Phase 1 temp→fsync→rename→row path (a real assembled file).
  - `x-amz-metadata-directive: COPY` (default) — carry the source's
    content-type and user metadata onto the copy.
  - `x-amz-metadata-directive: REPLACE` — take content-type and user metadata
    from the request instead, bytes unchanged.
  - **source==dest metadata-only update** — a copy where source key == dest key
    with `REPLACE` rewrites just the metadata/content-type row (the real SDK
    idiom CONCEPT calls out).
  - `CopyObjectResult` XML response: `ETag` + `LastModified`.
- **DeleteObjects** (`POST /<bucket>?delete`):
  - Delete a list of keys (up to 1000) in one request, each via the Phase 1
    row-first-then-unlink path.
  - `Deleted` result list; `Quiet` mode returns only errors.
  - Idempotent per key (a missing key still reports deleted), matching S3.

## Out of scope

- **No new UI, banner, or `/_/` surface** — presigned URLs are *generated by
  clients*, not by buckit. The presigned-URL **button** in the web UI is Phase 5
  (`/_/api/presign`); this phase only proves the S3-side of a URL a client
  already signed.
- **UploadPartCopy** (`x-amz-copy-source` on an UploadPart). Left
  `NotImplemented` as in Phase 3; promote only if a matrix client needs it.
- **Copy-source conditional headers** — `x-amz-copy-source-if-match` /
  `-if-none-match` / `-if-modified-since` / `-if-unmodified-since`. Parsed by
  `s3s`, accepted and ignored (CONCEPT non-goals; no matrix client depends on
  them for the happy path).
- **CORS / browser preflight** (`--cors`) — the presign button will tempt
  `fetch()` from a frontend, but CORS is explicitly a **v0.2** flag. A browser
  `GET`/`PUT` to a presigned URL still *works*; only cross-origin JS is gated,
  and that's out of scope here.
- **Versioning** — `versionId` on copy-source or in a DeleteObjects entry is
  ignored; there is one version of each key.
- **`x-amz-tagging-directive`, ACL/`x-amz-acl`, SSE/KMS, storage-class,
  checksum algorithms** on copy — CONCEPT non-goals; accepted and ignored.
- **DeleteObjects governance/bypass, MFA, object-lock** headers — non-goals.
- **Fixing the presigned host-in-signature Docker gotcha** — it is *inherent to
  SigV4* (the host is signed). We document it (one README paragraph, per
  CONCEPT's "known sharp edges"); we do not rewrite signatures.

## Behavior

### Presigned / query-string auth
- **Addressing:** path-style only, consistent with Phases 1–3.
- A request whose SigV4 arrives in the **query string** (`X-Amz-Signature`
  present) is authenticated exactly like a header-signed one: a correct
  signature over the canonical request succeeds; a wrong secret or a tampered
  path/query → **403 `SignatureDoesNotMatch`**.
- **Expiry:** `X-Amz-Expires` (seconds, relative to `X-Amz-Date`) is honored —
  a URL fetched after it lapses → **403 `AccessDenied`** (expired). A URL used
  within its window succeeds.
- **Payload:** presigned `PUT` uses `UNSIGNED-PAYLOAD` (SDKs don't hash the body
  into the signature); the body still streams through the normal Phase 1 write
  path and gets a correct content-MD5 ETag.
- **Host in signature (documented sharp edge):** the signed canonical request
  includes the `Host` header, so a URL signed for `localhost:9000` will
  `SignatureDoesNotMatch` if replayed against a different host:port (the classic
  Docker-Compose `localhost` vs. service-name gotcha). Behavior is correct S3;
  we only note it in docs.

### CopyObject
- **Copy source** arrives as `x-amz-copy-source: /<srcbucket>/<srckey>` (leading
  slash optional, URL-encoded); `s3s` parses it into the source bucket/key.
- **Resolution / errors (checked before any bytes move):**
  - Source bucket missing → **404 `NoSuchBucket`**.
  - Source key missing → **404 `NoSuchKey`**.
  - Destination bucket missing → **404 `NoSuchBucket`**.
- **Bytes:** copy the source file's bytes into `.tmp/` → fsync → atomic rename
  into `buckets/<destb>/<destkey>` (creating parents), then write the dest
  object row last (Phase 1 invariant: row is the source of truth). The result is
  one real browsable file.
- **ETag:** the copy **preserves the source object's ETag verbatim** — a
  single-part source keeps its hex-MD5, a multipart source keeps its composite
  `-N` ETag. (For identical bytes this equals a recomputed MD5 for single-part
  objects; preserving avoids re-hashing and keeps composite ETags honest. See
  open questions.)
- **Metadata directive:**
  - `COPY` (default): dest `content_type` and user metadata = the source row's.
  - `REPLACE`: dest `content_type` = request's (defaulting
    `binary/octet-stream` if absent, as in Phase 1) and user metadata =
    request's.
- **source==dest:**
  - With `REPLACE` → legal; rewrites the row's content-type/metadata,
    `last_modified = now`, bytes untouched (the metadata-only update idiom). May
    skip the byte copy since the file already holds the right bytes.
  - With `COPY` (no change) → **400 `InvalidRequest`** ("copy request is illegal
    … without changing the object's metadata, storage class, …"), matching S3,
    so SDKs that probe this get the expected error.
- **Overwrite:** copying onto an existing dest key replaces it (last writer
  wins), same `INSERT OR REPLACE` as Phase 1.
- **Response:** `CopyObjectResult` XML with the dest `ETag` and a
  `LastModified` = the copy time.

### DeleteObjects (batch)
- **Request:** `POST /<bucket>?delete` with an XML `<Delete>` body listing
  `<Object><Key>…</Key></Object>` entries (up to **1000**; more →
  **400 `MalformedXML`**/`InvalidRequest` per S3) and an optional
  `<Quiet>true</Quiet>`.
- **Per key:** row first, then unlink the file (Phase 1 delete path). A key with
  no object row still succeeds (idempotent) — S3 reports every requested key as
  deleted unless an actual error occurs.
- **Missing bucket:** `POST ?delete` against a nonexistent bucket →
  **404 `NoSuchBucket`** (whole request fails, no per-key results).
- **Response:** `DeleteResult` XML. Non-quiet → a `<Deleted><Key>…` entry per
  key. Quiet → only `<Error>` entries (empty result on full success). `s3s`
  builds the XML from the returned `DeleteObjectsOutput`.
- **VersionId:** present in an entry → ignored (no versioning); the current
  object is deleted and the key echoed in `Deleted`.

### Cross-cutting
- **Consistency:** SQLite stays the source of truth throughout — a copied object
  "exists" only once its row is written; a batch-deleted key is gone the moment
  its row is removed, unlink is best-effort cleanup (a `NotFound` unlink is
  ignored, as in Phase 1).
- **Listing/GET/HEAD interplay:** a copied object is an ordinary Phase 1/2
  object immediately (lists, Range-GETs, HEADs with its ETag); batch-deleted
  keys drop out of ListObjectsV2/v1 at once.

## Acceptance criteria

Named clients: **boto3** and **aws-sdk-js v3** (the two that generate presigned
URLs SDK apps actually use — aws-sdk-js is the browser-facing one), with the
**AWS CLI** for per-field assertions and copy/batch-delete, and filesystem
`cmp`/`Path` checks for the "real file" guarantees. Each becomes a plan
checkbox. (The full 5-client presigned matrix — adding aws-sdk-go-v2 and rclone
— is assembled in Phase 6; this phase proves the surface with the clients
drivable now.)

### Presigned URLs
- [ ] **boto3 presigned GET (the phase headline).** After putting
      `s3://b/hello.txt`, `boto3` `generate_presigned_url("get_object", …)` →
      fetching that URL with a **plain HTTP client (no credentials)** returns
      the exact bytes and `200`.
- [ ] **boto3 presigned PUT.** `generate_presigned_url("put_object", …)` →
      a credential-less `PUT` of a body to that URL returns `200`; a following
      authed `get_object` returns those bytes and
      `cmp s3data/buckets/b/<key>` is clean (filesystem assertion — real file
      landed via `UNSIGNED-PAYLOAD`).
- [ ] **aws-sdk-js v3 presigned GET.** `@aws-sdk/s3-request-presigner`
      `getSignedUrl` for a `GetObjectCommand` → `fetch` (no creds) returns the
      object bytes and `200` (proves a second, independently-signing SDK).
- [ ] **AWS CLI presign.** `aws s3 presign s3://b/hello.txt` → the printed URL
      fetched with `curl` returns the bytes and `200`.
- [ ] **Expired presigned URL → 403.** A URL signed with `--expires-in 1`
      (or `ExpiresIn=1`), fetched after it lapses → HTTP `403` with
      `AccessDenied` (expired), not the object.
- [ ] **Tampered presigned URL → 403.** Editing the signed path or a query param
      of a valid presigned URL → HTTP `403` `SignatureDoesNotMatch`, object not
      served.

### CopyObject
- [ ] **boto3 copy (`s3.copy_object`).** With `s3://b/src.bin` present,
      `copy_object(CopySource="b/src.bin", Bucket="b", Key="dst.bin")` → `200`;
      `get_object` on `dst.bin` returns the source bytes;
      `cmp s3data/buckets/b/dst.bin s3data/buckets/b/src.bin` is clean; the
      dest `ETag` equals the source `ETag`.
- [ ] **AWS CLI cross-bucket copy.** `aws s3 cp s3://b1/k s3://b2/k2` exits 0;
      `Path("s3data/buckets/b2/k2")` exists with the source bytes.
- [ ] **Metadata directive COPY carries source metadata.** Put `src` with
      `ContentType=application/json` and metadata `{"team":"x"}`; copy it to
      `dst` (default directive); `head_object("dst")` shows
      `ContentType=application/json` and `team=x`.
- [ ] **source==dest REPLACE (metadata-only update).** `copy_object` with
      `CopySource="b/k"`, `Bucket="b"`, `Key="k"`,
      `MetadataDirective="REPLACE"`, `ContentType="text/plain"`,
      `Metadata={"v":"2"}` → `200`; `head_object("k")` now shows
      `text/plain` and `v=2`, and `cmp s3data/buckets/b/k` still matches the
      original bytes (bytes untouched).
- [ ] **source==dest COPY → InvalidRequest.** `copy_object` with source key ==
      dest key and default (`COPY`) directive → `400 InvalidRequest` (S3's
      "illegal copy to itself" error).
- [ ] **Copy source missing → NoSuchKey; source bucket missing →
      NoSuchBucket.** `copy_object(CopySource="b/does-not-exist", …)` →
      `404 NoSuchKey`; `CopySource="no-bucket/k"` → `404 NoSuchBucket`.
- [ ] **Copy of a multipart object preserves the composite ETag.** Copy an
      object uploaded via Phase 3 multipart (ETag `"<hex>-N"`) → the dest
      `head_object` ETag is the same `"<hex>-N"`, and the dest bytes `cmp`
      clean against the source (confirms the ETag-preservation decision).

### DeleteObjects (batch)
- [ ] **boto3 batch delete.** Put `k1,k2,k3`; `delete_objects(Delete={"Objects":
      [{"Key":"k1"},{"Key":"k2"},{"Key":"k3"}]})` → response `Deleted` lists all
      three; `Path("s3data/buckets/b/k1")` etc. are all **gone**; a following
      `list_objects_v2` returns none of them.
- [ ] **Batch delete is idempotent.** Include a never-existed key
      (`{"Key":"ghost"}`) in the batch → it appears in `Deleted` (or at least
      does not error), and the request succeeds `200`.
- [ ] **Quiet mode.** `delete_objects(Delete={"Objects":[…],"Quiet":true})` on
      all-valid keys → response has **no `Deleted` entries** and no `Errors`
      (empty success), keys still removed from disk.
- [ ] **`aws s3 rm --recursive` (batch under the hood).** After putting several
      keys under a prefix, `aws s3 rm s3://b/prefix/ --recursive` exits 0 and
      `ls s3data/buckets/b/prefix/` shows the files gone (proves the CLI's batch
      path works end-to-end).

## Resolved decisions

All confirmed by the user — no open questions block planning.

- **CopyObject ETag: preserve vs. recompute.** ✅ **Preserve the source ETag
  verbatim.** For single-part sources this equals a recomputed content-MD5, so
  it's indistinguishable; for multipart sources it keeps the composite `-N`
  ETag, which is cheap (no re-hash) and matches how sync tools identify the
  object. Revisit only if a matrix client asserts a *recomputed* single-part
  ETag after copying a multipart object (none in the matrix are known to).
- **Copy byte path: `fs::copy` vs. stream-through-`.tmp`.** ✅ **Stream through
  `.tmp/` → fsync → rename**, reusing the Phase 1 write path for crash-safety
  and the single atomic-rename invariant, rather than a bare `fs::copy` onto the
  live key. The source==dest `REPLACE` case may **skip the copy** entirely
  (bytes already correct — row-only update).
- **Is any presigned code actually needed?** Expectation is **no** — `s3s` +
  `SimpleAuth` already validate query-string SigV4. The first plan step is a
  spike that drives a presigned GET; if it already passes, the "verification"
  boxes are the deliverable and no handler changes ship for auth. If a gap
  surfaces, the plan closes it minimally. (Flagged so the plan doesn't assume
  work that may not exist.)
- **DeleteObjects >1000 keys.** ✅ Reject a batch over 1000 with
  `400 MalformedXML`/`InvalidRequest` (S3's bound). Cheap guard; keeps us honest
  against clients that probe the limit. `s3s` may already enforce this at
  parse time — confirm in the plan and don't double-implement.
- **Per-key error reporting granularity.** ✅ For a dev tool, treat the only
  realistic per-key failure (an internal unlink/DB error) as a request-level
  `500`; the happy path reports every requested key in `Deleted`. We don't
  fabricate per-key `Error` entries for missing keys — S3 reports those as
  deleted too.
